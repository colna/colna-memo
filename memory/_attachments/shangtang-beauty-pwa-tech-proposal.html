§ 0 · Value

核心业务价值验证:对方能看到美颜吗?

这是方案的核心业务价值,不是加分项。答案是能,因为咱们改的是上行流本身,不是本地滤镜。

做法对方能看到吗说明
❌ 本地 CSS 滤镜 / 只改 <video> 显示 看不到 上行推流用的还是原始摄像头帧,只是本地渲染多加了一层 filter
✅ 改上行流本身(咱们方案) 一定看得到 商汤处理后的 canvas → captureStream() → MediaStreamTrack → TRTC 推流,推的就是美颜后的帧

对方零负担

美颜全在主播侧算,对方拿到已美颜好的视频帧,不需装 SDK、不耗对方 CPU、带宽不变。

2

对方 App 不用改

只主播端 PWA 集成一次,对方无论 PWA / iOS / Android / Web 只要能收 TRTC 流就能看。

3

主播自拍所见即所得

本地预览和上行流是同一份 canvas,主播看到什么效果对方就看到什么,零落差。

§ 2 · Context

目标产品

sitin-next app-pwa 是 heyhru 的 PWA 端(H5 女主播端,视频通话主要发生在这里)。视频通话走腾讯云 TRTC。当前视频通话没有美颜,女主播上镜前必须自己修图,产品需要通话内实时美颜/滤镜/贴纸能力。

参考实现

商汤 shangtang-sdk-demo 已拿到 SenseMARS WebEffects 2.3 授权(license 到 2026-07-26),里面 8 类 SDK 全部本地可跑,可作集成参考。

现有栈评估

框架React 19 + Vite + Zustand,和商汤 demo Vue 3 不同,但纯浏览器 SDK 不受框架约束
视频通话trtc-sdk-v5 ^5.15.2 + tuicall-engine-webrtc ^3.1.7

§ 3 · Options

三条技术路径对比

TRTC v5 官方内置两个美颜插件 + 支持自定义视频源,合计三条路径。

路径 A · 兜底

TRTC BasicBeauty(官方内置)

能力:magnitude / brightness / ruddy 三项基础美颜
  • 成本:TRTC v5 内置,不用商汤,不用授权
  • 短板:能力单一,没有细分美妆 / 滤镜 / 贴纸
  • 定位:SDK 加载失败 / 弱机时的降级兜底
路径 B · 官方高级

TRTC Beauty(付费包)

能力:whiten / dermabrasion / lift / shave / eye / chin + Effect[] 滤镜贴纸
  • 成本:需腾讯单独购买(sdkAppId + userId + userSig 授权)
  • 短板:能力薄于商汤,没 11 类细分美妆 / 染发 / 美甲 / AR 试戴
  • 定位:不做主选
★ 路径 C · 推荐

自定义 videoTrack + 商汤 SDK

能力:商汤 4 类 SDK 完整特效栈
  • effects:美颜 / 美形 / 微整形 / 滤镜 / 贴纸 / 整妆 / 背景虚化
  • makeup:口红 · 唇线 · 眼影 · 眼线 · 睫毛 · 腮红 · 修容 · 粉底 · 眉毛 …
  • hair 染发 · nail 美甲
  • 成本:商汤 license(已有,需商务续期 + OSS 分发路径)
i
结论:路径 C 是唯一能覆盖产品完整需求的方案。路径 A 作为降级兜底(SDK 加载失败 / 弱机时退回 BasicBeauty,不影响通话)。

4.3 SDK 加载策略

!
商汤单个 SDK ≈ 2-5MB(js + wasm + data),4 类合计 15-20MB。不能首屏加载。
  • 登录后 requestIdleCallback 里只预热 effects(最常用)。
  • makeup / hair / nail 懒加载:用户点击「细节美妆」时才 import()。
  • 通话开始时若 SDK 未加载完,先降 BasicBeauty 上行,后台继续加载,完成后 updateLocalVideo({option.videoTrack: newTrack}) 热切换。

4.4 License 管理

  • dev:public/sdk/*/WebAR.lic 直接放本地(商汤 demo 就是这么用)。
  • test / prod:走自有 OSS(需商汤商务开通,demo 的 cdf-tryon.oss-cn-hangzhou.aliyuncs.com/sdk/softsugar/* 是 softsugar 客户的,不能直接用)。
  • 代码里 makeup/hair/nail 走 checkLicenseFromPath(sdkPath) 显式校验,effects/tryon 隐式校验。
  • 续期:到期前 7 天(2026-07-19)找许佳续,同步升级到 2.4+ 如有。

4.6 兼容 TUICallEngine

TUICallEngine.accept() 内部走 TRTC.enterRoom,视频推流由 TUICallEngine.openCamera(viewId) 触发。接自定义流:

  • 不能同时用 openCamera 和 trtc.startLocalVideo —— 前者会覆盖后者。
  • 方案:accept() 之后跳过 openCamera,直接 this.trtc.startLocalVideo({view, publish: true, option: {videoTrack}}),view 传本地预览 DOM ID。
  • 只 hack webCallManager.startLocalVideo 分叉,不改 TUICallEngine 或 TRTC 库本身。
§ 5 · Downlink

下行美颜:给主播看到的对方加美颜(对方无感知)

独立可选扩展。ROI 明显低于上行,建议只做「环境增强档」,不做「改脸档」。

5.1 场景定义

上行美颜(§0-4)主播摄像头帧被商汤 SDK 处理后再推流 → 对方看到美颜后的主播 · 对方能看到
下行美颜(本节)主播 SDK 收到对方视频后,在主播端本地做处理 → 主播 UI 上渲染的是美颜后的对方;对方 App/CPU/带宽完全不变,对方无感知

5.2 技术可行性:一手证据核验

✓
TRTC v5 官方原生支持,不需要任何 hack。关键 API:trtc.getVideoTrack({ userId }) 直接返回远端 MediaStreamTrack,可塞给商汤 SDK 处理。

核验证据(直接来自本仓库已装 SDK 类型定义,不是搜索或记忆):
文件 sitin-next/node_modules/trtc-sdk-v5/index.d.ts:2216 · SDK 版本 5.15.3-beta.9(与 app-pwa/package.json 声明的 ^5.15.2 一致) 

// 官方注释原文:
// If not passed or passed an empty string, get the local videoTrack.
// Pass the userId of the remote user to get the remote user's videoTrack.

getVideoTrack(config?: {
  userId?: string;              // 传对方 userId 拿远端 track
  streamType?: TRTCStreamType;  // MAIN 摄像头 / SUB 屏幕共享,都能拿
  processed?: boolean;           // v5.8.2+,是否要 SDK 处理后的 track
}): MediaStreamTrack | null;

顺带核验到:getAudioTrack 同样支持传 remote userId(index.d.ts:2207),意味着「下行音频降噪 / 变声」技术路径也通,虽然本方案不做。

5.5 产品分级建议 — 关键:技术能做 ≠ 产品该做

档 A · 推荐做

环境增强档

背景虚化 bokeh · 亮度/曝光补偿 · 时域降噪
  • 男用户环境昏暗 / 房间乱 / 光线差 → 让主播看清对方,减少视觉疲劳
  • 弱网 480p 下行画质差 → 降噪缓解
  • 不改脸,不影响主播对对方真实长相的判断
档 B · 不推荐

改脸档

磨皮 · 美白 · 瘦脸 · 眼线 · 唇线 · 贴纸
  • 下行 480p → 商汤关键点精度掉,精细美妆基本不能用
  • 主播看到的对方被美化 → 线下/其他场景暴露真实长相 → 情感判断错位(heyhru 讲主播粘性,这个落差有风险)
  • 双向都开美颜,GPU/CPU 翻倍,弱机掉帧 ×2
  • ROI 弱:「让男用户顺眼」→「主播接单」因果链条弱
→
最终建议:
P0 — 只做上行(主播美颜)。
P1 — 追加下行 A 档「环境增强」(bokeh + 亮度 + 降噪),对主播实际体验有帮助且无落差风险。
P2 — 产品坚持时才考虑下行 B 档「改脸」,只开磨皮/美白,回避精度问题。
