## 工作日志 — sp-follow 错误日志诊断 + dom_snapshot 截断放宽
**时间** 14:30
**做了什么** 分析 sp-follow 错误日志 CSV(06-18 ~ 06-26,3911 条),定位 686 条 `Cannot read properties of undefined (reading 'execute')` 根因:`GraceChat-Earn-Android` 仓库 `haven/.../ActionDispatcher.kt:108-119` 在 WebView 注入 `window.SocialProxy` 完成前就调 `window.SocialProxy.execute(...)`,catch 后塞裸字符串到 `error` 字段上传 → server `device.gateway.ts:339` 走 `JSON.stringify(string)` 写入 message 列。1743 条 ELEMENT_NOT_FOUND 里 91% 实际是 404 / 登录墙 / 5xx 误归类(detectPageState 06-25 才上线,且 `title === "Instagram"` 精确匹配漏掉 `(9+) Instagram` 角标 title)。
**为什么** 用户排查 sp-follow 错误大盘想知道根因分布。诊断时发现 `context_html` 截 2000 字符,IG body 几百 KB,看不到结构;candidates outerHTML 截 300 字符,IG 长 className + style 全被切。
**结果** sitin-next commit `529b50cf`(personal/zz/sp-follow) → push 完成。改 `packages/app-ins-scripts/src/instagram/actions/utils.js` 8 处 `substring(0, N)` 统一从 50/100/120/300/500/2000 放宽到 10000。元素数量上限(candidates × 5、visible_buttons × 20)保持不变。
**坑** Android WebView 注入时序竞态产出的错误形态是「裸 `e.message` 字符串」而不是 `{code, message}` object;诊断时如果只在 sitin-next 仓库 grep `.execute(` 调用会找不到 bug(穷举完都不可能 undefined)。真正 bug 在 APK 仓库,不在脚本仓库。
**沉淀** 沉淀到 `30-Resources/troubleshooting/sitin-next-script-error.md`(三条:execute 时序竞态、ELEMENT_NOT_FOUND 误归类、INVALID_PARAMS 上游 caller bug)。
