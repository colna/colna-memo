## 工作日志 — 04:51 修复 colna sync 顺序 + autostash

- **做了什么**:改 `colna-memo/src/gitsync.rs` 的 `run_sync`:调整为「先 add+commit memory/ → 再 git pull --rebase → reindex 一次 → push」,并给 pull 加 `--autostash`。
- **为什么**:原顺序 pull 在 commit 之前,工作区有未提交改动时 `git pull --rebase` 会被拒(error: cannot pull with rebase: You have unstaged changes),导致 pull 被静默跳过、多设备无法对齐真源。
- **结果**:`cargo build` 通过。先提交本地再 pull,工作区干净;`--autostash` 进一步兜底任何残留未提交改动(如源码改动),rebase 后自动恢复。reindex 由两次合并为一次(commit+pull 后状态稳定再建索引)。
- **关键命令 / diff**:`gitsync.rs` run_sync 步骤重排;pull 改为 `["pull","--rebase","--autostash"]`。
- **沉淀**:见 `30-Resources/troubleshooting/colna-sync.md`。

## 工作日志 — 05:19 sitin-next UploadTest 接入 OSS 上传

- **做了什么**:sitin-next 切到 `feature/admin` 分支;在 `packages/app-minerva-web/src/pages/UploadTest/index.tsx` 新增「OSS 上传」按钮,接入 minerva-server 的 `POST /api/upload/oss/signed-url`(签名直传)。
- **为什么**:页面原本只有 GCS 直传(服务端中转),需补充阿里云 OSS 上传入口。
- **结果**:eslint 通过(0 error / 0 warning)。改动文件仅 `UploadTest/index.tsx`(从 feature/sp 切到 feature/admin,工作区原本干净)。GCS 按钮保留,新增 OSS 按钮并列;上传成功复用 `result` state 展示 `cdnUrl` + size。**sitin-next 改动未提交**(按规则不自动 commit)。
- **关键点**:
  - OSS 与 GCS 模式不同:GCS 用 antd `<Upload action>` 服务端中转;OSS 是**签名直传三步**(取 signed-url → 前端 `PUT` 直传 OSS → 用 cdnUrl),用 antd `customRequest` 实现。
  - **坑**:`contentType` 必须在「取签名」和「PUT 上传」两处完全一致,否则 OSS 拒绝;故统一 `rcFile.type || 'application/octet-stream'`。
  - 接口返回 `{ success:true, data:{ uploadUrl, objectKey, cdnUrl } }`(以 controller 为准,文档里写的 `{code:0}` 已过时)。
  - 照搬项目既有范式:`import type { RcFile } from 'antd/es/upload'` + `UploadProps['customRequest']`(参考 `Lovia/components/ImageUploader.tsx`);把上传逻辑抽成组件外 `uploadFileToOss` 避免 `max-lines-per-function` warning。
- **沉淀**:OSS 签名上传前端接入模式,见 `30-Resources/instagram-script-selector-conventions.md` 同目录可另起一篇(暂记于此)。
