### 07:30 — 导出体积实测 + 前端改流式写盘(不再缓内存)

- 背景:实测每行 CSV 平均 ~9KB(`domSnapshot.target_html` 占大头,单行最大 23KB)。7天 4.2万行 ≈360MB、全量 14万行 ≈1.2GB。`target_html` 是脚本失败瞬间目标 DOM 的 HTML 快照(排错看选择器为何失配的核心证据),**用户明确不能从 CSV 去掉**。
- 问题:服务端流式无上限,但 07:08 版前端是 `responseType:'blob'` 把整份缓进浏览器内存——360MB 勉强、1.2GB 会崩标签页(且 >512MB 单 JS string 会 `Invalid string length`,blob 装配不受此限但仍占 RAM)。
- 改法(纯前端,后端不动):`app-minerva-web/.../SocialProxy/api.ts` 把 `exportScriptErrorLogs`(blob)换成 `streamExportScriptErrorLogs`——用 **File System Access API**(`showSaveFilePicker` → `createWritable`)+ 原生 `fetch`,`res.body.pipeTo(writable)` **分块直写磁盘,内存恒定**,GB 级也能导;auth 用 `getSessionToken()` 拼 Bearer + `getMinervaServerUrl()` 拼 URL(绕开 umi wrapper 拿原始流);不支持 FSA API 的浏览器(Firefox/Safari)回退 blob 下载。`ScriptErrors/index.tsx` 的 `handleExport` 改为调它,`AbortError`(用户取消保存框)不报错。
- 验证:前端两文件 `eslint`/`tsc` 0 error。
- 沉淀:浏览器导大文件别 `await res.blob()`(整缓 RAM,>2GB 崩、字符串 >512MB throw);要无上限就 `showSaveFilePicker` + `res.body.pipeTo(createWritable())` 流式落盘,内存与文件大小无关。代价:FSA API 仅 Chromium;内部后台是 Chrome,可接受 + 留 blob 回退。

### 07:46 — 导出进度:Modal + MB 计数 + 可取消
